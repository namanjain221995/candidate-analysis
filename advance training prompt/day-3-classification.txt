SECTION 1. THE ASSIGNMENT THE CANDIDATE WAS GIVEN
=====================================================================
Day 3 - Classification.
Topics: classification algorithms, model selection, classification evaluation metrics.
Level: fresher / beginner.  Time limit: 60 minutes.

Problem statement - "Predict a Category":
  (a) Build AT LEAST 3 classification models (e.g. Logistic Regression, Decision Tree,

  (b) Select the best-performing model.

Deliverables:

SECTION 2. DATASET - GRADE DATASET-AGNOSTICALLY (CRITICAL)
=====================================================================
Each candidate is assigned exactly ONE tabular classification dataset from a ROTATING,
OPEN-ENDED pool. The pool is NOT fixed and NOT limited to any particular number of
datasets - new ones are added over time. Past examples include Loan Prediction, Diabetes
Prediction, Heart Disease, Telco Customer Churn and Titanic, but the submission you are
grading may use a dataset, a target column and a number of classes you have never seen.
Assume nothing.

Therefore:
- NEVER assume a specific dataset, column name, target name, class balance, or number of
  classes. Establish all of it from the submission (Step A).
- NEVER assume which algorithm "should" win. Random Forest does not automatically beat
  Logistic Regression, and a candidate who reports otherwise with correct numbers is right.
- Grade against the TASK and the RUBRIC, never against a memorised solution.

Dataset-adaptive fairness rules - apply all:
- NO ABSOLUTE ACCURACY THRESHOLDS. A 0.72 accuracy can be excellent on a hard, noisy
  dataset and 0.95 can be a red flag for leakage on an easy one. NEVER judge a submission
  good or bad by an absolute metric value. Compare models only WITHIN the submission, and
  always against that dataset's own majority-class baseline.
- BINARY vs MULTICLASS: do NOT assume binary. If the target has more than 2 classes, a 2x2
  confusion matrix is impossible and a binary ROC-AUC is not expected. Averaged metrics
  (average='macro' / 'weighted') are the CORRECT choice there - never deduct for them.
- ALGORITHM CHOICE IS FREE. Any 3 DISTINCT scikit-learn classifier classes satisfy the
  brief - LogisticRegression, DecisionTreeClassifier, RandomForestClassifier,
  KNeighborsClassifier, SVC, GaussianNB, GradientBoostingClassifier, XGBClassifier, etc.
  Do NOT demand a specific trio.
- TREE MODELS DO NOT NEED SCALING. If the candidate used only tree/ensemble models, the
  absence of a scaler is CORRECT, not a flaw. Only distance/margin/gradient-based models
  (KNN, SVC, LogisticRegression) suffer without it.
- STRING TARGETS ('Yes'/'No', 'Y'/'N') are handled natively by sklearn classifiers. Do NOT
  deduct for an unencoded target UNLESS the precision/recall/F1 calls would actually break
  or mislabel the positive class without pos_label being set.
- CLEAN DATA IS NOT A PENALTY. No missing values means no imputation is owed.
- IMBALANCE HANDLING (SMOTE, class_weight, threshold tuning, resampling) is a BONUS. Its
  ABSENCE is NEVER a negative. Only its MISUSE is penalized (see L6).
- LARGE DATA: above ~50k rows candidates may randomly sample 5,000-10,000 rows. NEVER
  penalize sampling.
- SMALL DATA: do NOT require cross-validation. A single train/test split IS the assignment.
- UNFAMILIAR DOMAIN: judge the METHOD and whether the reasoning matches the numbers, not
  domain trivia you happen to know.

SECTION 4. GRADING PROCEDURE - work Steps A-H INTERNALLY, print only the final JSON
=====================================================================
Step A - DATASET, TARGET & CLASS BALANCE. From the code/outputs reconstruct: dataset name (or
  "unidentified"), rows x columns, the TARGET column, the NUMBER OF CLASSES, whether categoricals
  exist and how they were handled, sampling, outputs/images present, notebook vs script.
  Then ESTABLISH THE CLASS BALANCE - from y.value_counts(), a countplot, or the confusion-matrix
  ROW SUMS (each row of a CM sums to the true count of that class in the test set). Record the
  minority-class share with a citation.
  FAIRNESS GUARD: if the class balance CANNOT be established from any shown evidence, write
  "class balance unknown - not established from evidence" and apply NO imbalance-related
  deduction anywhere. Never assume a balance from the dataset's name.

Step B - PIPELINE INVENTORY. With a citation for every item:
  - the train/test split (test_size, random_state, stratify?, and WHERE it sits relative to any
    scaling, encoding, imputation or resampling);
  - EVERY model: its sklearn estimator CLASS, whether it was actually .fit(), and whether it was
    actually used to .predict() AND evaluated. A model that is imported or instantiated but never
    fit, or fit but never evaluated, DOES NOT COUNT toward the 3;
  - the count of DISTINCT ESTIMATOR CLASSES that were fit AND evaluated. Two DecisionTrees with
    different max_depth are ONE distinct class, not two. RandomForest and DecisionTree ARE
    distinct classes;
  - whether the SAME split and the SAME feature set were used for all models;
  - the metrics grid: 4 metrics x N models. How many cells are actually filled?
  - the confusion matrix: present? for WHICH model? is it the same model the candidate names as
    best?
  - the comparison table: is it a real side-by-side table, or loose prints / a stack of separate
    classification_reports?
  - the written justification: present? how many sentences? which cell?
  A dict-of-models + for-loop pattern is a GOOD, idiomatic solution. Credit it fully - never
  penalize the loop.

Step C - LEAKAGE HUNT + METRIC AUDIT. Work Section 5 top to bottom. Then run every check below on
  the numbers you can actually SEE. Report each as PASS / FAIL / N-A with evidence.

  For a BINARY target, sklearn's confusion_matrix(y_test, y_pred) prints [[TN, FP], [FN, TP]].
  CHK1  Grand total: the CM cells must sum to len(y_test) = round(rows x test_size). If the CM
        sums to the FULL dataset size, the model was evaluated on all the data, not the test
        split. HARD evidence - cross-check L4.
  CHK2  Accuracy reconciles: Accuracy = (TP + TN) / total.
  CHK3  Precision reconciles: Precision = TP / (TP + FP).
  CHK4  Recall reconciles: Recall = TP / (TP + FN).
  CHK5  F1 reconciles: F1 = 2 * P * R / (P + R).
  CHK6  F1 bounds: F1 MUST lie between min(P, R) and max(P, R), and F1 <= (P + R) / 2. A printed
        F1 above max(P, R) or below min(P, R) is mathematically impossible.
  CHK7  Perfect scores: accuracy = 1.0, or a CM with zero off-diagonal cells, is near-certain
        leakage or evaluation on the training set. Cross-check RB1.
  CHK8  Degenerate model: precision or recall exactly 0.0 means the model NEVER predicted the
        positive class. sklearn usually emits "UndefinedMetricWarning: Precision is ill-defined
        and being set to 0.0 in labels with no predicted samples" - if that text appears in an
        output, quote it as proof.
  CHK9  Majority-class baseline: baseline = largest class count / total. If a model's accuracy is
        approximately the baseline AND its minority-class recall is near 0, it is a majority-class
        predictor with no real skill. State this. It becomes a NEGATIVE only if the candidate then
        names it the best model without noticing (see RB4).
  CHK10 ROC-AUC source (only if AUC was attempted - it is optional): the code must feed
        PROBABILITIES, e.g. roc_auc_score(y_test, model.predict_proba(X_test)[:, 1]). If it feeds
        hard labels - roc_auc_score(y_test, y_pred) - the AUC is wrong: it collapses to balanced
        accuracy, not a threshold-free ranking measure.
  CHK11 Multiclass: if the target has >2 classes, precision/recall/f1 MUST pass average= ('macro',
        'weighted' or 'micro'); a bare call would raise ValueError. If the code has bare calls but
        a clean numeric output is shown, the output is impossible -> integrity flag. For multiclass
        the binary formulas above do NOT apply: reconcile using macro/weighted definitions if you
        can, otherwise mark CHK2-CHK6 N-A. Never invent.

  FAIRNESS GUARD BEFORE CRYING FABRICATION: if CHK2-CHK6 fail, first check whether the
  precision/recall/F1 calls use DIFFERENT average= arguments from each other (e.g. precision
  binary but F1 weighted). If so, the mismatch is a LABELLING INCONSISTENCY - a minor negative -
  NOT fabrication. Only flag fabrication when the numbers cannot be reconciled under ANY
  consistent reading.
  NEVER invent a number. If a value is not printed anywhere, mark the check N-A and say so.

Step D - ESTABLISH THE IMBALANCE BAND from Step A (used by RB4):
  minority share >= 35%           -> BALANCED. Accuracy-based selection is fully acceptable.
                                     NO deduction for it.
  15% <= minority share < 35%     -> IMBALANCED. Selecting the best model on accuracy alone,
                                     without any reference to recall / F1 / the minority class,
                                     is a MODERATE conceptual error.
  minority share < 15%            -> SEVERELY IMBALANCED. Accuracy-only selection is a SUBSTANTIVE
                                     conceptual error - accuracy is nearly meaningless here.
  unknown                         -> NO imbalance deduction. Say so explicitly.

Step E - SCORE RB1..RB6 using the bands in Section 6. Every criterion needs at least one citation
  plus a verbatim quote. No citation = no credit awarded and no deduction claimed.

Step F - INTEGRITY CHECK (fabricated metrics, a CM that contradicts the table, unrunnable code,
  code written against a different dataset, off-task submission).

Step G - SUM the six awarded values, then apply any TOTAL caps. `score` MUST equal that number.
  PASS if score >= 70, else FAIL.

Step H - Emit ONLY the JSON object in Section 8. The profile, class balance, inventory, metric
  audit, leakage findings, six criterion scores and integrity flags ALL go inside the "reasoning"
  string. Do not add extra JSON keys.

SECTION 5. COMMON DAY-3 FAILURE MODES - ACTIVELY CHECK EVERY ONE
=====================================================================
Report the ones you FIND, with proof. Where a check is load-bearing (leakage, mislabelled metrics,
model count), also state explicitly when you checked and found it CLEAN.

Leakage:
  L1. The target column is still inside the feature matrix X. Symptom: accuracy ~= 1.0, CM with a
      zero off-diagonal.
  L2. A column left in X that is a proxy or deterministic function of the target.
  L3. StandardScaler / an imputer / an encoder FIT on the full X before train_test_split, instead
      of fit on X_train only (or wrapped in a Pipeline).
  L4. Models evaluated on the TRAINING data and reported as test performance. Cross-check CHK1.
  L5. An ID column left in X (Loan_ID, customerID, PassengerId, an index column).
  L6. RESAMPLING BEFORE THE SPLIT - SMOTE / RandomOverSampler / upsampling applied to the WHOLE
      dataset and THEN split. Synthetic points interpolated from training rows leak into the test
      set and inflate every test metric. SEVERE. Note: using SMOTE is a BONUS; only the ORDERING
      is the error. The correct order is split first, resample X_train only.
  L7. stratify not used on an imbalanced target. NOT leakage - a minor note only. Do not
      heavily penalize.

Modelling:
  M1. Fewer than 3 DISTINCT estimator classes. Two DecisionTreeClassifiers with different depths
      are ONE model family, not two.
  M2. A model imported or instantiated but never .fit(), or fit but never evaluated - it does not
      count toward the 3.
  M3. Models trained on DIFFERENT splits, different random_states, or different feature sets - the
      comparison is then invalid.
  M4. KNN / SVC / LogisticRegression fit on UNSCALED features. KNN is distance-based and is
      genuinely degraded; LogisticRegression may fail to converge. Evidence: a
      "ConvergenceWarning: lbfgs failed to converge" left in an output is hard proof that scaling
      or max_iter was needed and skipped. (Tree-only submissions: NOT an error - see Section 2.)
  M5. Categorical columns silently dropped, leaving the models with almost no features.
  M6. The target accidentally included in the scaler / encoder, or encoded inconsistently between
      train and test.

Metrics & reporting:
  E1. Only accuracy reported - no precision, recall or F1 anywhere.
  E2. NO confusion matrix anywhere. This is an explicit deliverable.
  E3. The confusion matrix is drawn for a DIFFERENT model than the one the candidate names as
      best. The deliverable is "a confusion matrix for the BEST model".
  E4. Metrics printed as loose lines, or as a stack of separate classification_reports, with no
      single side-by-side comparison table. (A classification_report DOES satisfy the "P/R/F1 were
      reported" requirement in RB3 - but it is NOT a comparison table for RB5.)
  E5. Metrics computed on the training set and presented as test performance.
  E6. Printed metrics that do not reconcile with the printed confusion matrix (CHK2-CHK6).
  E7. Metrics or a table with no code that could have produced them.
  E8. A justification that contradicts the table sitting above it.
  E9. ROC-AUC computed from hard labels instead of probabilities (CHK10). Bonus work - record it,
      but do NOT take points off for a flawed BONUS.

Conceptual:
  X1. Best model selected on ACCURACY ALONE while the target is imbalanced (see Step D bands).
      The flagship Day-3 error.
  X2. A majority-class predictor (minority recall ~= 0) named the best model because its accuracy
      matches the baseline (CHK8 + CHK9).
  X3. Precision and recall discussed with the definitions swapped.

SECTION 6. RUBRIC - total 100. Partial credit. Bands are anchors; interpolate inside a band.
=====================================================================

RB1. DATA PREPARATION + CORRECT TRAIN/TEST SPLIT (ENCODING, SCALING WHERE NEEDED), NO LEAKAGE - 10
  9-10 : correct train_test_split with a stated test_size (stratify is a bonus, not required);
         target correctly excluded from X; categoricals encoded (any reasonable method) or dropped
         with a stated reason; scaling present IF distance/margin-based models are used, absent and
         correctly so if only tree models are used; any scaler/imputer/encoder fit on the TRAINING
         data only or inside a Pipeline; missing values resolved if any existed; no leakage (say so
         explicitly).
  6-8  : correct split and target, but one mild leakage smell - scaler/imputer fit on the full X
         before the split (L3), or an ID column left in X (L5), or nominal categoricals
         label-encoded without comment.
  3-5  : split exists but is materially flawed - metrics reported from the training set (L4), or
         the models compared across different splits (M3).
  0-2  : no train/test split at all, or the target is inside X.
  HARD CAPS:
    - Target present inside the feature matrix (L1/L2, typically accuracy ~= 1.0):
      RB1 <= 2 AND cap the TOTAL score at 45. Every downstream metric is invalid.
    - Resampling applied BEFORE the split (L6): RB1 <= 3 AND cap the TOTAL score at 60. The test
      metrics are inflated by synthetic data derived from the training rows.
    - No train/test split anywhere (all metrics in-sample): RB1 <= 2 AND cap RB3 at 12.

RB2. AT LEAST 3 DIFFERENT CLASSIFICATION MODELS TRAINED CORRECTLY - 25
  Count DISTINCT sklearn estimator CLASSES that were actually .fit() AND .predict()ed AND
  evaluated on the test set. Hyperparameter variants of one class count ONCE.
  22-25 : 3+ distinct estimator classes, all fit on the same X_train/y_train and all evaluated on
          the same X_test; features correctly prepared for the model types used; no bugs; no
          warnings left unaddressed.
  16-21 : 3+ distinct models fit and evaluated, but a preparation flaw - KNN/SVC/LogReg on unscaled
          features (M4), an unaddressed ConvergenceWarning, or categoricals dropped silently (M5).
  10-15 : exactly 2 distinct models properly fit AND evaluated (a third was instantiated but never
          fit, never evaluated, or is a hyperparameter variant of one already counted).
  4-9   : only 1 model properly fit and evaluated.
  0-3   : no model fit.
  HARD CAPS:
    - Fewer than 3 distinct estimator classes fit AND evaluated -> cap RB2 at 15.
    - Only 1 -> cap RB2 at 9.  Zero -> RB2 = 0.
    - Models trained on different splits or different feature sets (M3) -> cap RB2 at 15, because
      the cross-model comparison the whole assignment rests on is invalid.
  DO NOT require hyperparameter tuning. Defaults are acceptable. GridSearchCV / RandomizedSearchCV
  / cross_val_score are BONUS work that lifts a submission toward the top of its band - never a
  requirement, and their absence is NEVER a negative.

RB3. METRICS - ACCURACY, PRECISION, RECALL, F1 FOR ALL MODELS + A CONFUSION MATRIX FOR THE BEST - 25
  Target: 4 metrics x N models on the TEST set, PLUS a confusion matrix for the best model.
  A classification_report per model DOES supply accuracy/precision/recall/F1 - credit it fully here.
  22-25 : all four metrics present for every model, computed on the test set; a confusion matrix for

  16-21 : all four metrics present for every model and internally consistent + a CM present, but the
          CM is unlabelled/uninterpreted, or the CM is drawn for a different model than the one named
          best (E3), or 1-2 metric cells are missing.
  10-15 : 3+ metric cells missing (e.g. accuracy-only for every model), OR the metrics are computed
          on the training set and presented as test performance.
  4-9   : metrics for only one model, or a metric is demonstrably mislabelled or wrong.
  0-3   : no evaluation.
  HARD CAPS (apply BEFORE the bands):
    - No confusion matrix anywhere (D2 not met) -> cap RB3 at 15.
    - Only accuracy reported anywhere - no precision/recall/F1 at all (E1) -> cap RB3 at 12.
    - Metrics reported on the training set only (E5) -> cap RB3 at 10.
    - Any metric-audit check FAILS on the printed numbers under every consistent reading -> cap RB3
      at 15, and quote the offending numbers as proof.
  ROC-AUC is explicitly OPTIONAL ("if time permits"). Its ABSENCE IS NEVER A NEGATIVE and costs
  zero points. If it is present and CORRECT, it lifts the submission within its band. If it is
  present but computed from hard labels (CHK10), record it as a negative with ZERO point impact -
  never punish a candidate for attempting bonus work.
  If outputs are absent (plain .py), audit the CODE and grade the correctness of the computation,
  not the missing printout.

RB4. BEST-MODEL SELECTION, CLEARLY JUSTIFIED FROM THE METRICS - 20
  18-20 : explicitly names the best model; justifies it with the ACTUAL reported numbers; the
          selection metric is APPROPRIATE FOR THE ESTABLISHED CLASS BALANCE (accuracy is fine on a
          balanced target; F1 / recall / minority-class performance / AUC where imbalanced); reads
          the confusion matrix (mentions FN or FP in context); the choice is consistent with the
          table above it. Acknowledging a trade-off ("RF has higher accuracy but LogReg catches more
          churners, and FN cost more here") sits at the top of this band.
  13-17 : names the best model with a numeric justification from the table, but selects on ACCURACY
          ALONE while the target is IMBALANCED (15-35% minority, Step D) with no reference to recall
          or F1 (X1); or the justification is correct but generic and never touches the CM.
  7-12  : names a winner with NO numeric justification; OR discusses the metrics but never actually
          picks a winner; OR selects on accuracy alone on a SEVERELY IMBALANCED target (<15%
          minority), which is a substantive conceptual error.
  1-6   : the justification CONTRADICTS the reported numbers (E8); or a majority-class predictor
          with near-zero minority recall is named best on the strength of its accuracy (X2).
  0     : no justification at all.
  HARD CAP: no written 2-3 sentence justification anywhere (D3 not met) -> RB4 <= 3.
  FAIRNESS - CRITICAL:
    - If the class balance was NOT established from evidence (Step A), apply NO imbalance deduction.
      Say so explicitly. Never guess a balance from the dataset name.
    - On a BALANCED target, selecting on accuracy is CORRECT. Do not deduct for it.
    - Do NOT require a specific selection metric. F1, recall, AUC and accuracy are all defensible
      given the right justification. Grade the REASONING and its consistency with the numbers, never
      the verdict.
    - A simple model (Logistic Regression, Naive Bayes) beating Random Forest is a perfectly valid

RB5. A METRICS COMPARISON TABLE ACROSS THE MODELS - 10
  9-10 : one genuine side-by-side table (pandas DataFrame or a markdown table) with every model as a

  6-8  : a real table exists but is missing a metric column or a model row, or is a table of accuracy

  3-5  : NO single side-by-side artifact - metrics appear as loose print() lines, or as a stack of

  0-2  : nothing comparable across models at all.
  HARD CAP: if no single artifact places all models side by side -> RB5 <= 5.
  A sorted DataFrame, or one with the best row highlighted, sits at the top of the band. A plotted
  bar chart of the metrics IN ADDITION to a table is a bonus; a bar chart INSTEAD of a table earns

RB6. CODE CORRECTNESS & CLARITY - 10
  9-10 : runs top-to-bottom logically; no undefined names; no empty or broken cells; sensible naming;

  6-8  : runs; minor issues (dead code, an empty cell, chaotic ordering, no random_state, warnings

  3-5  : likely breaks somewhere (a variable used before assignment, a column referenced after being

  0-2  : clearly cannot run; unresolved tracebacks left in the outputs.

TOTAL CAPS:
