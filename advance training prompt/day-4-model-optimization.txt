SECTION 1. THE ASSIGNMENT THE CANDIDATE WAS GIVEN
=====================================================================
Day 4 - Model Optimization.
Topics: cross-validation, hyperparameter tuning, GridSearchCV, RandomizedSearchCV,
ensemble learning.
Level: fresher / beginner.  Time limit: 60 minutes.

Problem statement - "Tune & Ensemble a Classifier":
  (a) Take a BASELINE classifier (e.g. Decision Tree or Logistic Regression).
  (b) Evaluate it with 5-fold Cross Validation.
  (c) Tune it with GridSearchCV AND with RandomizedSearchCV.
  (d) Build ONE ensemble model (Random Forest, Bagging or Voting Classifier).
  (e) Compare all results in a single table.

Deliverables:
  D1. Notebook showing the baseline CV score, and GridSearchCV + RandomizedSearchCV
      best params / best scores.
  D2. One ensemble model trained AND evaluated on the SAME split.
  D3. A single results table: Model | CV Accuracy | Test Accuracy | Best Params.

Notation below: RB1..RB6 = the rubric criteria. CHK<n> = the audit checks in Step C.

WHAT DAY 4 DOES *NOT* ASK FOR - read this before you deduct anything:
  - NO written conclusion or justification paragraph is a deliverable. Its absence is
    NEVER a negative. Commentary on which model won is a BONUS.
  - NO precision / recall / F1 / confusion matrix / ROC-AUC is a deliverable. The table
    columns are literally named CV Accuracy and Test Accuracy. ACCURACY IS THE
    BRIEF-SPECIFIED METRIC FOR THIS DAY. Do NOT import Day-3 reflexes and do NOT deduct
    for accuracy-only reporting, even on an imbalanced target. Reporting other metrics,
    or tuning with scoring='f1' / 'roc_auc', is a BONUS.
  - NO data cleaning / EDA is a deliverable (that was Day 1). Only the preparation needed
    to make the models fit is in scope.

SECTION 2. DATASET - GRADE DATASET-AGNOSTICALLY (CRITICAL)
=====================================================================
Each candidate is assigned exactly ONE tabular classification dataset from a ROTATING,
OPEN-ENDED pool. The pool is NOT fixed and NOT limited to any particular number of
datasets - new ones are added over time. Past examples include Breast Cancer Wisconsin,
Pima Indians Diabetes, Heart Disease Risk, Adult Census Income and Telco Customer Churn,
but the submission you are grading may use a dataset, a target column and a number of
classes you have never seen. Assume nothing.

Therefore:
- NEVER assume a specific dataset, column name, target name, class balance, or number of
  classes. Establish all of it from the submission (Step A).
- NEVER assume which model "should" win, or that tuning "should" improve on the baseline.
- Grade against the TASK and the RUBRIC, never against a memorised solution.

Dataset-adaptive fairness rules - apply all:
- NO ABSOLUTE ACCURACY THRESHOLDS. 0.74 can be excellent on a hard dataset and 0.99 can be
  a leakage red flag on an easy one. NEVER judge a submission good or bad by an absolute
  score. Compare only WITHIN the submission.
- TUNING MAY LEGITIMATELY FAIL TO HELP. sklearn defaults are often near-optimal, and on a
  small grid in 60 minutes the tuned CV score frequently ties or slightly loses to the
  baseline. "GridSearchCV gave no improvement here" is a CORRECT, honest finding. NEVER
  penalize it. Only penalize a candidate who CLAIMS an improvement the numbers contradict.
- THE ENSEMBLE MAY LEGITIMATELY LOSE to the tuned baseline. Never penalize that verdict.
- SMALL GRIDS ARE CORRECT. This is a 60-minute task. A grid of 4-8 candidates over 2-3
  hyperparameters is FULL credit. NEVER demand a large search space, exotic parameters, or
  n_estimators sweeps. A search that would take 20 minutes to run is a design flaw, not a
  strength.
- BINARY vs MULTICLASS: do NOT assume binary. On a multiclass target, accuracy still works
  and is still the required column - nothing changes.
- IMBALANCE HANDLING (class_weight, SMOTE, stratify) is a BONUS. Its ABSENCE is NEVER a
  negative on Day 4. Only its MISUSE is penalized (see L8).
- TREE MODELS DO NOT NEED SCALING. If the baseline and ensemble are tree/forest models, the
  absence of a scaler is CORRECT, not a flaw. Only KNN / SVC / LogisticRegression need it.
- CLEAN DATA IS NOT A PENALTY. No missing values means no imputation is owed.
- LARGE DATA: above ~50k rows candidates may randomly sample 5,000-10,000 rows. NEVER
  penalize sampling - on Day 4 it is actively sensible, since GridSearchCV on 50k rows will
  not finish inside the time limit.
- SMALL DATA: cv=5 as specified is enough. Do NOT demand nested CV or repeated CV.
- UNFAMILIAR DOMAIN: judge the METHOD and whether the reasoning matches the numbers, not
  domain trivia you happen to know.

Step C - LEAKAGE HUNT + AUDIT. Work Section 5 top to bottom. Then run every check below on the
  numbers and code you can actually SEE. Report each as PASS / FAIL / N-A with evidence.

  CHK1  Fold count: if a per-fold score array is printed (cross_val_score returns one), it must
        contain exactly `cv` values. A 5-fold CV printing 3 or 7 numbers contradicts the code.
  CHK2  CV mean reconciles: the reported baseline CV accuracy must equal the mean of the printed
        fold scores, within the printed rounding. COMPUTE IT. A mean that does not match the folds
        is fabricated or copied from elsewhere.
  CHK3  CV bounds: the reported mean must lie between the min and the max of the printed folds, and
        every accuracy must lie in [0, 1].
  CHK4  Grid candidate count: N_GRID = the product of the lengths of the lists in param_grid. If
        verbose output is present, GridSearchCV prints "Fitting <cv> folds for each of <N_GRID>
        candidates, totalling <cv * N_GRID> fits". All three numbers must reconcile with the grid
        SHOWN IN THE CODE. A mismatch means the printed output did not come from that grid ->
        integrity flag. (If verbose was not enabled, mark N-A - this is common and is NOT a flaw.)
  CHK5  Randomized fit count: RandomizedSearchCV with verbose prints "Fitting <cv> folds for each of
        <n_iter> candidates, totalling <cv * n_iter> fits". Must match the n_iter in the code.
  CHK6  n_iter vs space size: if param_distributions are discrete LISTS, n_iter must not exceed
        N_SPACE, or sklearn emits "The total space of parameters <N> is smaller than n_iter=<M>.
        Running <N> iterations. For exhaustive searches, use GridSearchCV." If that text appears in
        an output, quote it - it means the "random" search was exhaustive and pointless.
  CHK7  best_params_ IS A MEMBER OF THE DECLARED SPACE. Every key in best_params_ must be a key in
        the param_grid / param_distributions, and every VALUE must be one of the values offered for
        that key. A best_params_ containing max_depth=12 when the grid only offered [3, 5, 7] is
        HARD PROOF the output did not come from this code -> integrity flag. Do this check for BOTH
        searches. It is the strongest fabrication detector on Day 4.
  CHK8  best_score_ IS A CV SCORE, NOT A TEST SCORE. Check where the Test Accuracy column comes
        from. If the tuned rows' Test Accuracy equals their CV Accuracy to every decimal, or the
        code literally writes best_score_ into a column headed "Test Accuracy", then NO held-out
        performance was ever measured. This is a substantive conceptual error, not a typo.
  CHK9  Tuned-vs-baseline direction: if the tuned best_score_ is BELOW the baseline CV score, the
        grid did not contain the baseline's default configuration. That is LEGAL and common - do
        NOT deduct for it. It becomes a NEGATIVE only if the candidate simultaneously CLAIMS that
        tuning improved the model (a claim the numbers contradict).
  CHK10 Scoring/label match: if the search was built with scoring='f1' / 'roc_auc' / anything other
        than accuracy, then best_score_ is NOT an accuracy, and placing it in a column headed
        "CV Accuracy" mislabels it. Note: CHOOSING a non-accuracy scorer is GOOD practice and is
        rewarded - only the column LABEL is the error.
  CHK11 Perfect scores: accuracy 1.0, or a CV mean of 1.0, is near-certain leakage or in-sample
        evaluation. Cross-check L1 and L6.
  CHK12 Same-split check: every Test Accuracy in the table must come from the SAME X_test. If the
        code calls train_test_split more than once, or with different random_state values, or
        re-splits before the ensemble, the Test Accuracy column is not comparable across rows.
  CHK13 cv_results_ consistency (only if a cv_results_ DataFrame is printed): max(mean_test_score)
        must equal best_score_, and the row at best_index_ must carry best_params_.
  CHK14 Table completeness: expected rows = baseline, Grid-tuned, Randomized-tuned, ensemble = 4.
        Expected data columns = CV Accuracy, Test Accuracy, Best Params. Count the filled cells out
        of 12. A Best Params cell reading "default" or "-" for the baseline/ensemble is CORRECT and
        COMPLETE - do not count it as missing.

  NEVER invent a number. If a value is not printed anywhere, mark the check N-A and say so.
  FAIRNESS GUARD BEFORE CRYING FABRICATION: rounding differences, a mean printed to 2 decimals
  against folds printed to 4, or a random_state that makes exact reproduction impossible are NOT
  fabrication. Only flag fabrication when the numbers cannot be reconciled under ANY consistent

=====================================================================
SECTION 5. COMMON DAY-4 FAILURE MODES - ACTIVELY CHECK EVERY ONE
=====================================================================
Report the ones you FIND, with proof. Where a check is load-bearing (leakage, degenerate searches,
a fake Test Accuracy column), also state explicitly when you checked and found it CLEAN.

Leakage:
  L1. The target column is still inside the feature matrix X. Symptom: accuracy ~= 1.0.
  L2. A proxy or deterministic function of the target left in X.
  L3. Scaler / imputer / encoder FIT on the full X BEFORE train_test_split, instead of fit on
      X_train only (or wrapped in a Pipeline). Real leakage into the test set. MODERATE.
  L4. *** THE FLAGSHIP DAY-4 ERROR *** GridSearchCV / RandomizedSearchCV fit on the FULL X, y
      instead of X_train, y_train, while a "Test Accuracy" is STILL reported from X_test. The test
      rows sat inside the CV folds that selected the hyperparameters, AND best_estimator_ was
      refit on all of X including X_test. The reported test accuracy is in-sample and worthless.
      SEVERE -> TOTAL cap.
      SUB-CASE L4b: the search is fit on the full X, y and NO separate test accuracy is reported at
      all. This is NOT leakage - there is simply no held-out evaluation. Do NOT apply the severe
      cap. Score it as a missing Test Accuracy column in RB5 and note the methodology.
  L5. The BASELINE cross_val_score run on the full X, y while the TUNED models were searched on
      X_train. The CV column is then not comparable row-to-row, and the baseline CV number saw the
      test rows. MINOR-MODERATE - it does not corrupt the test accuracies, only the CV column.
  L6. A model scored on the TRAINING data and reported as Test Accuracy (.score(X_train, y_train)
      sitting under a "Test Accuracy" heading).
  L7. An ID column left in X.
  L8. Resampling (SMOTE / RandomOverSampler) applied to the whole dataset BEFORE the split, so
      synthetic points derived from training rows leak into the test set. SEVERE. Note: USING SMOTE
      is a bonus; only the ORDERING is the error.

  *** NOT AN ERROR - DO NOT DEDUCT *** Scaling X_train once, before passing it to cross_val_score
  or to a search, instead of wrapping the scaler in a Pipeline so it refits per fold. This is
  technically fold-level leakage, but it is FRESHER-ACCEPTABLE and standard in every tutorial.
  Using a Pipeline is a BONUS that lifts a submission toward the top of its band. Its ABSENCE IS
  NEVER A NEGATIVE. Do not raise it, do not deduct for it.

Searching:
  M1. DEGENERATE GRID - every list in param_grid has length 1, so exactly ONE candidate is
      searched. GridSearchCV then does nothing but a CV of a single fixed configuration.
  M2. THIN GRID - one hyperparameter with 2-3 values. Legitimate but minimal. Middle band, not a cap.
  M3. GridSearchCV and RandomizedSearchCV run on DIFFERENT base estimators (e.g. a Decision Tree
      tuned with Grid and a Random Forest tuned with Randomized). The brief says take a baseline and
      tune IT with both, so the two searches are no longer comparable and the baseline was never
      actually tuned twice. MODERATE.
  M4. n_iter = 1, or a RandomizedSearchCV space with a single configuration - the search is vacuous.
  M5. n_iter >= the size of a discrete space, so the random search is exhaustive and slower than a
      grid (CHK6). MINOR - note it, deduct at most 2.
  M6. The tuned best_estimator_ is never scored on X_test, so the tuned rows have no Test Accuracy.
  M7. cv is not 5, deviating from the explicit brief. MINOR (1-3 points inside RB1/RB2/RB3), never a
      cap. StratifiedKFold(n_splits=5) IS 5-fold and is correct - credit it as equivalent or better.
  M8. Nothing is ever CV'd - a single train/test score is presented as a "CV score". The defining
      technique of Day 4 is absent.

Ensembling:
  N1. NO ensemble anywhere.
  N2. The "ensemble" is the SAME estimator class as the baseline (baseline = RandomForest,
      "ensemble" = RandomForest) - no new model was introduced.
  N3. The ensemble is fit but never evaluated on the test set, or is evaluated on a DIFFERENT split
      / a different random_state, breaking the "same split" requirement.
  N4. A VotingClassifier built with a single estimator, or with voting='soft' on a member lacking
      predict_proba (which would raise at runtime - if a clean output is nonetheless shown, that is
      an integrity flag).
  ACCEPT AS AN ENSEMBLE, without argument: RandomForestClassifier, BaggingClassifier,
  VotingClassifier, ExtraTreesClassifier, AdaBoostClassifier, GradientBoostingClassifier,
  HistGradientBoostingClassifier, StackingClassifier, XGBoost / LightGBM. Do NOT demand a specific
  one. Do NOT require the ensemble to be TUNED - defaults are fine; tuning it is a BONUS.

Reporting:
  E1. best_params_ not printed for GridSearchCV.        E2. best_score_ not printed for GridSearchCV.
  E3. Either not printed for RandomizedSearchCV.
  E4. NO single results table - the numbers exist only as scattered print() lines.
  E5. The table is missing a required column (most often Best Params).
  E6. The Test Accuracy column populated with best_score_ (a CV number), so no held-out result was
      ever reported (CHK8). SUBSTANTIVE.
  E7. Fewer than 4 model rows in the table.
  E8. A table with no code that could have produced it.
  E9. A written claim that tuning improved the model, contradicted by the numbers in the table
      (CHK9). (Note: writing NO commentary at all is fine - it is not a Day-4 deliverable.)

SECTION 6. RUBRIC - total 100. Partial credit. Bands are anchors; interpolate inside a band.
=====================================================================
NOTE ON SCOPE: this rubric deliberately has NO standalone data-preparation criterion - Day 4 is
about optimization, not cleaning. Data preparation and the train/test split are judged ONLY insofar
as they corrupt the results: through the leakage deductions inside RB1-RB4 and the TOTAL caps below.

RB1. BASELINE CLASSIFIER EVALUATED WITH 5-FOLD CROSS-VALIDATION - 15
  A "baseline" means an UNTUNED, default-parameter, NON-ensemble estimator. Using defaults IS the
  point of a baseline - never deduct for it.
  13-15 : a clear baseline classifier instantiated and evaluated with 5-fold CV (cross_val_score /
          cross_validate / StratifiedKFold(n_splits=5)); the per-fold scores AND/OR their mean
          printed; run on X_train (or on the full X where no separate test claim is made); features
          adequately prepared for that estimator (categoricals encoded; scaled if LogReg/KNN/SVC).
  10-12 : baseline CV run correctly, but k != 5 (M7), OR only the mean shown with no fold detail,
          OR the baseline CV was run on the full X while the searches used X_train (L5), making the
          CV column not comparable row-to-row.
  6-9   : a baseline model is fit and scored, but with a SINGLE train/test evaluation and NO
          cross-validation anywhere (M8); OR CV was run only on a tuned/ensemble model, never on a
          plain untuned baseline.
  3-5   : baseline instantiated but never fit, or never scored.
  0-2   : no baseline at all.
  HARD CAP: no cross-validation anywhere in the submission -> RB1 <= 6. Cross-validation is the
    defining technique of the day.
  cross_val_score on a classifier already defaults to StratifiedKFold. Writing it out explicitly
  with shuffle=True and a random_state is a BONUS, never a requirement.

RB2. GRIDSEARCHCV RUN CORRECTLY, WITH BEST PARAMS AND BEST SCORE REPORTED - 25
  22-25 : GridSearchCV over a REAL grid (>= 2 hyperparameters and >= 4 total candidate
          configurations); cv=5; fit on X_train ONLY; BOTH best_params_ AND best_score_ printed;
          the tuned estimator subsequently scored on X_test (via best_estimator_, via grid.score(),
          or by refitting with **best_params_ - all three are valid and equally correct); every
          applicable audit check passes.
  17-21 : the search runs and both best_params_ and best_score_ are reported, but the grid is THIN
          (one hyperparameter, or 2-3 candidates total) (M2), OR cv != 5 (M7), OR the tuned
          estimator is never scored on X_test (M6) so its Test Accuracy cell is empty.
  11-16 : only ONE of best_params_ / best_score_ is reported; OR the search was fit on the FULL X, y
          while a Test Accuracy is still claimed from rows inside that X (L4 - also triggers the
          TOTAL cap below).
  5-10  : GridSearchCV is present but DEGENERATE - every parameter list has length 1, so exactly one
          candidate configuration is "searched" (M1) and no tuning actually occurred.
  0-4   : no GridSearchCV anywhere.
  HARD CAPS:
    - GridSearchCV absent -> RB2 = 0.
    - Degenerate grid, exactly 1 candidate configuration -> cap RB2 at 10.
    - Neither best_params_ nor best_score_ printed -> cap RB2 at 12.
    - CHK7 fails (best_params_ contains a value not offered anywhere in the declared grid) -> cap
      RB2 at 12, quote the offending value, and raise an integrity flag.
  DO NOT demand a large grid. 60 minutes. Four candidates over two hyperparameters is FULL credit.
  A grid so large it could not have run in the time limit is a design flaw, not a strength.

RB3. RANDOMIZEDSEARCHCV RUN CORRECTLY, WITH BEST PARAMS AND BEST SCORE REPORTED - 20
  18-20 : RandomizedSearchCV over a real param_distributions with an explicit n_iter >= 5; on the
          SAME base estimator as the GridSearchCV run (the brief says tune the baseline with BOTH);
          cv=5; fit on X_train only; BOTH best_params_ AND best_score_ printed; audit checks pass.
          Sampling from scipy distributions (randint / uniform) is a BONUS - list-based
          param_distributions are perfectly acceptable and cost nothing.
  14-17 : runs and both reported, but n_iter is tiny (2-4), OR n_iter >= the size of a discrete space
          so the search is exhaustive and pointless (M5, CHK6), OR the tuned estimator is never
          scored on X_test (M6), OR the space is byte-identical to the GridSearchCV grid (defensible
          as a direct method comparison - deduct at most 2, and only if nothing else offsets it).
  8-13  : only ONE of best_params_ / best_score_ reported; OR run on a DIFFERENT base estimator than
          the Grid run (M3), so the two searches are not comparable and the baseline was not actually
          tuned with both methods; OR fit on the full X, y while a Test Accuracy is still claimed (L4).
  3-7   : present but degenerate (n_iter=1, or a single-configuration space) (M4).
  0-2   : no RandomizedSearchCV anywhere.
  HARD CAPS:
    - RandomizedSearchCV absent -> RB3 = 0.
    - Neither best_params_ nor best_score_ printed -> cap RB3 at 10.
    - CHK7 fails on this search -> cap RB3 at 10, quote the offending value, raise an integrity flag.

RB4. ONE ENSEMBLE MODEL TRAINED AND EVALUATED ON THE SAME SPLIT - 20
  18-20 : an ensemble DISTINCT from the baseline, .fit() on the SAME X_train and .score()/.predict()
          on the SAME X_test as every other model; a CV score also produced for it, so its CV
          Accuracy cell in the table can be filled; no bugs.
  14-17 : ensemble fit and test-scored on the same split, but no CV score for it (leaving its CV
          Accuracy cell empty); or a VotingClassifier with a questionable configuration (N4).
  8-13  : ensemble fit but evaluated on a DIFFERENT split / different random_state (N3), so its Test
          Accuracy is not comparable to the rest of the table; OR fit but never evaluated on the test
          set at all.
  4-7   : the "ensemble" is the SAME estimator class as the baseline (N2) - no new model was
          introduced; OR instantiated but never fit.
  0-3   : no ensemble anywhere.
  HARD CAPS:
    - No ensemble -> RB4 = 0.
    - Ensemble evaluated on a different split from the other models -> cap RB4 at 13. "The same
      split" is an explicit part of the deliverable.
    - No train/test split anywhere, so "evaluated on the same split" is impossible -> cap RB4 at 10.
  DO NOT require the ensemble to be tuned - the brief says "build one Ensemble model". Defaults are
  fine. Tuning it is a BONUS.
  DO NOT penalize the ensemble for losing to the tuned baseline. That is a legitimate, common result.

RB5. A SINGLE RESULTS TABLE: MODEL | CV ACCURACY | TEST ACCURACY | BEST PARAMS - 10
  The target is ONE artifact with a row per model - baseline, Grid-tuned, Randomized-tuned, ensemble
  (4 rows) - and the four named columns.
  9-10 : one genuine side-by-side table (pandas DataFrame or a markdown table) with all four columns
         and all four model rows; every cell filled; CV Accuracy and Test Accuracy are genuinely
         different quantities (a CV mean and a held-out score respectively). A Best Params cell
         reading "default" or "-" for the baseline and the ensemble is CORRECT and COMPLETE - it is
         NOT a missing cell.
  6-8  : a real single table exists but one COLUMN is missing (most often Best Params), or one model
         ROW is missing, or a few cells are empty.
  3-5  : NO single artifact - the numbers appear only as loose print() lines scattered across cells
         and the reader must assemble the comparison themselves (E4); OR the Test Accuracy column is
         populated from best_score_ (a CV number), so the table has two CV columns and no held-out
         result anywhere (E6 / CHK8).
  0-2  : nothing comparable across models at all.
  HARD CAPS:
    - No single side-by-side artifact -> RB5 <= 5.
    - Test Accuracy column filled with best_score_ / CV values -> RB5 <= 5, AND record it as a
      substantive negative: no held-out performance was ever measured for the tuned models.
  A bar chart IN ADDITION to the table is a bonus. A bar chart INSTEAD of the table earns the 6-8
  band at most - the deliverable specifies a table.

RB6. CODE CORRECTNESS & CLARITY - 10
  9-10 : runs top-to-bottom logically; no undefined names; no empty or broken cells; sensible naming;
         brief comments or markdown structure. A dict-of-models loop, or a list-of-dicts assembled
         into the results DataFrame, is idiomatic and earns full marks.
