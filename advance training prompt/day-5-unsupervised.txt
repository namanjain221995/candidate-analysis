SECTION 1. THE ASSIGNMENT THE CANDIDATE WAS GIVEN
=====================================================================
Day 5 - Unsupervised Learning.
Topics: clustering, K-Means, DBSCAN, PCA.
Level: fresher / beginner.  Time limit: 60 minutes.

Problem statement - "Segment the Data":
  (a) SCALE the features.
  (b) Apply K-MEANS, using the ELBOW METHOD to pick k.
  (c) Apply DBSCAN to the SAME data.
  (d) COMPARE the groupings the two algorithms produce.
  (e) Use PCA to reduce the data to 2 dimensions and VISUALIZE the clusters from BOTH
      algorithms on a scatter plot.

Deliverables:
  D1. Notebook with scaling, an elbow-method plot, K-Means and DBSCAN clustering.
  D2. PCA-reduced 2D scatter plots colored by cluster, for BOTH algorithms.
  D3. 3-4 sentences describing what each cluster represents.

Notation below: RB1..RB6 = the rubric criteria. CHK<n> = the audit checks in Step C.
WCSS = within-cluster sum of squares = KMeans .inertia_.

*** WHAT DAY 5 DOES *NOT* ASK FOR - READ THIS BEFORE YOU DEDUCT ANYTHING ***
This is UNSUPERVISED learning. Do NOT import reflexes from the supervised days.
  - THERE IS NO TRAIN/TEST SPLIT, NO TARGET AND NO ACCURACY. Fitting the scaler, PCA,
    K-Means and DBSCAN on the FULL dataset is CORRECT AND EXPECTED. It is NOT leakage.
    NEVER deduct for "the scaler was fit on all the data" or "there is no held-out set".
    Do not raise it, do not mention it as a weakness.
  - NO supervised model, no accuracy / precision / recall / F1 / confusion matrix / ROC.
  - NO hyperparameter search (GridSearchCV / RandomizedSearchCV) - that was Day 4.
  - NO cleaning-and-EDA report - that was Day 1. Only the preparation needed to make the
    clustering RUN is in scope, and it is scored inside RB1.
  - BONUS-ONLY, NEVER REQUIRED, ABSENCE IS NEVER A NEGATIVE: silhouette score,
    Davies-Bouldin / Calinski-Harabasz, a k-distance (NearestNeighbors) plot to choose eps,
    3D PCA, t-SNE / UMAP, hierarchical clustering, dendrograms, cluster naming/personas,
    validating clusters against a true label column if the dataset happens to have one.
    A FLAW IN BONUS WORK COSTS ZERO POINTS.

SECTION 2. DATASET - GRADE DATASET-AGNOSTICALLY (CRITICAL)
=====================================================================
Each candidate is assigned exactly ONE tabular dataset from a ROTATING, OPEN-ENDED pool.
The pool is NOT fixed and NOT limited to any particular number of datasets - new ones are
added over time. Past examples include Mall Customer Segmentation, Wine for Clustering,
Wholesale Customers, and credit-card customer datasets, but the submission you are grading
may use a dataset, a feature set and a row count you have never seen. Assume nothing.

Therefore:
- NEVER assume a specific dataset, column name, feature count, or "correct" number of
  clusters. Establish all of it from the submission (Step A).
- NEVER assume what k "should" be. There is NO right answer for k. k=3 and k=5 can both be
  defensible on the same data. Grade the JUSTIFICATION, never the value.
- NEVER assume DBSCAN "should" find a particular number of clusters.
- Grade against the TASK and the RUBRIC, never against a memorised solution.

Dataset-adaptive fairness rules - apply all:
- NO ABSOLUTE THRESHOLDS ON ANY CLUSTERING NUMBER. Inertia values, silhouette scores, noise
  percentages and explained-variance ratios are all dataset-dependent. A silhouette of 0.28
  can be honest and normal; 0.9 can be an artifact. NEVER judge a submission good or bad by
  the magnitude of an unsupervised score.
- THERE IS NO GROUND TRUTH. The clusters cannot be "wrong". Only the METHOD can be wrong.
- A WEAK OR AMBIGUOUS ELBOW IS A REAL, COMMON OUTCOME. Many datasets produce a smooth WCSS
  curve with no sharp elbow. A candidate who says "the elbow is not sharp; k=4 is the best
  compromise" is being HONEST and CORRECT. NEVER penalize an ambiguous elbow - only penalize
  a k that is never tied to the plot at all.
- LOW EXPLAINED VARIANCE IS A REAL RESULT. If PC1+PC2 capture only 35% of the variance, the
  2D picture is genuinely lossy. Saying so is a STRENGTH, not a weakness. Never deduct for a
  low explained-variance ratio.
- ALREADY-COMPARABLE FEATURES: if every column is already on the same scale (e.g. all are
  1-5 ratings, or the source file ships pre-standardized), the damage from skipping scaling
  is much smaller. Note it, deduct inside RB1, but do NOT apply the UNSCALED TOTAL CAP.
- CATEGORICAL / ID / TEXT COLUMNS: Euclidean clustering needs a numeric matrix. Dropping an
  ID column and dropping or one-hot-encoding categoricals are BOTH acceptable. Do not demand
  a specific approach.
- LABEL COLUMN, IF THE DATASET HAS ONE: using it to VALIDATE the clustering afterwards
  (crosstab, adjusted_rand_score) is excellent BONUS work. Using it as a clustering INPUT
  FEATURE is an error - the algorithm is then partly clustering on the answer (see P3).
- LARGE DATA: above ~50k rows candidates may randomly sample 5,000-10,000 rows. NEVER
  penalize sampling - DBSCAN is O(n^2)-ish in the worst case and sampling is sensible here.
- ANY SCALER IS FINE: StandardScaler, MinMaxScaler, RobustScaler, Normalizer. Do not demand
  a specific one.
- UNFAMILIAR DOMAIN: judge the METHOD and whether the interpretation matches the numbers,
  not domain trivia you happen to know.

SECTION 4. GRADING PROCEDURE - work Steps A-H INTERNALLY, print only the final JSON
=====================================================================
Step A - DATASET & FEATURE-MATRIX PROFILE. From the code/outputs reconstruct: dataset name (or
  "unidentified"); rows x columns of the RAW file; which columns were DROPPED (IDs, categoricals,
  a label column) and which were KEPT; whether missing values existed and how they were handled;
  whether sampling was used; and then the SHAPE OF THE MATRIX ACTUALLY PASSED TO THE CLUSTERING -
  call this n_used x p_used. Record it with a citation; you need it for CHK2. Also record: which
  scaler, whether outputs and figures are present, notebook vs script.

Step B - PIPELINE INVENTORY. With a citation for every item:
  - SCALING: which scaler, fit on what, and does the scaled matrix flow into K-Means, DBSCAN AND
    PCA - or only into some of them?
  - ELBOW: is there a loop over k? Over what RANGE (call it K_RANGE)? Does it collect .inertia_?
    Are the inertia values PRINTED anywhere, or only plotted? Is the elbow PLOT rendered? Is it
    axis-labelled?
  - K-MEANS: the final n_clusters (call it K_FINAL). Is K_FINAL inside K_RANGE? Is the choice
    JUSTIFIED in text by reference to the plot? Fit on the scaled matrix? Are labels produced and
    are cluster sizes reported?
  - DBSCAN: eps and min_samples (explicit, or defaults). Fit on THE SAME scaled matrix as K-Means?
    Are the number of clusters (EXCLUDING noise) and the noise-point count reported?
  - COMPARISON: are the two groupings actually compared - a crosstab, a side-by-side count, or a
    written comparison? Or were they merely both run?
  - PCA: n_components. Fit on the SCALED matrix? Used for the plots? Is explained_variance_ratio_
    reported?
  - PLOTS: is there a PC1-vs-PC2 scatter coloured by the K-MEANS labels? Is there one coloured by
    the DBSCAN labels? (Two subplots in one figure counts as both.) Are they titled/labelled? Is
    DBSCAN noise visually distinguished?
  - INTERPRETATION: is there written text describing what the clusters represent? How many
    sentences? Is there a cluster-profile table (groupby on the label) behind it?
  A loop over k, or a dict of models, is idiomatic and GOOD. Never penalize the loop.

Step C - PIPELINE HUNT + AUDIT. Work Section 5 top to bottom. Then run every check below on the
  numbers, code and FIGURES you can actually see. Report each as PASS / FAIL / N-A with evidence.

  CHK1  INERTIA IS MONOTONE. KMeans WCSS is mathematically NON-INCREASING in k: more centroids can
        never increase the within-cluster sum of squares. If a printed inertia list RISES as k
        rises, or the elbow plot visibly slopes UPWARD, the numbers did not come from this code ->
        integrity flag. (A tiny wobble at large k from a local minimum is possible; only flag a
        clear, material increase.)
  CHK2  *** THE INERTIA ANCHOR - THE STRONGEST FABRICATION DETECTOR ON DAY 5 ***
        IF StandardScaler was applied to all p_used columns and all n_used rows were passed to
        KMeans, THEN sklearn's StandardScaler uses POPULATION variance, so every scaled column
        satisfies sum((x - mean)^2) = n_used exactly. Therefore:
          - inertia at k=1 MUST equal approximately n_used * p_used;
          - and for EVERY k, inertia(k) <= n_used * p_used.
        Compute n_used * p_used from Step A and compare it to the printed inertia values, or to
        the y-axis of the elbow plot at its leftmost point. If the k=1 inertia is materially off
        (say, more than ~10% away), or ANY printed inertia EXCEEDS n_used * p_used, then the
        curve did not come from this data -> integrity flag. Quote the numbers.
        SUB-CASE, MinMaxScaler: every column lies in [0,1], so its variance is at most 0.25 and
        inertia(k) <= 0.25 * n_used * p_used for every k. Use that as the bound instead.
        Mark N-A if: a different scaler was used, the loop does not include k=1 AND no inertia
        value can be read, the matrix shape was never printed, or the inertia values are not
        readable from either an output or the plot. NEVER guess n_used or p_used.
  CHK3  K_FINAL IS INSIDE K_RANGE. The n_clusters passed to the final KMeans must be one of the k
        values the elbow loop actually swept. A final n_clusters=12 when the loop ran range(1,11)
        could not have been read off that plot -> the elbow did not inform the choice.
  CHK4  THE STATED ELBOW MATCHES THE CODE. If the candidate writes "the elbow is at k=4" and the
        code fits n_clusters=6 (or vice versa), the justification contradicts the model that was
        actually built. Quote BOTH.
  CHK5  KMEANS LABEL COUNT. sklearn's KMeans always returns exactly K_FINAL distinct, non-empty
        labels (0..K_FINAL-1); it relocates empty clusters. So if cluster sizes / value_counts are
        printed, the number of distinct labels MUST equal K_FINAL. Four distinct labels from an
        n_clusters=5 fit is impossible -> integrity flag.
  CHK6  LABEL COUNTS SUM TO n_used. For both algorithms, the printed cluster sizes must sum to
        n_used (the row count of the matrix actually clustered). If they sum to something else, the
        labels came from a different matrix than the code builds.
  CHK7  DBSCAN LABEL SEMANTICS. DBSCAN emits -1 for NOISE and 0..m-1 for clusters. The cluster
        count must therefore be len(set(labels)) - (1 if -1 in labels else 0). If the candidate
        counts len(set(labels)) directly, the reported cluster count is INFLATED BY ONE and the
        noise points are being presented as a cluster - a real conceptual error. Also: a printed
        DBSCAN label set that skips 0, or contains a value below -1, is impossible.
  CHK8  DBSCAN DEGENERACY. Read the label counts (or the scatter). If ~100% of points carry -1,
        DBSCAN found NO clusters. If a single non-noise label holds ~100% of points, DBSCAN found
        ONE cluster and produced no segmentation at all. Either way the DBSCAN scatter is one flat
        colour and any "interpretation" of it is vacuous. Record which case applies - it drives
        Step D and RB3.
  CHK9  SILHOUETTE FEASIBILITY (bonus work - a flaw here costs ZERO points, but a fabricated value
        is still fabrication). Any printed silhouette_score must lie in [-1, 1]. And
        silhouette_score RAISES "ValueError: Number of labels is 1. Valid values are 2 to
        n_samples - 1" when the labels have a single distinct value - so a clean silhouette number
        shown alongside an all-noise DBSCAN result is an IMPOSSIBLE output -> integrity flag.
  CHK10 PCA VARIANCE RATIOS. If explained_variance_ratio_ is printed: its length must equal
        n_components; every entry must lie in [0,1]; the entries MUST be in DESCENDING order (PCA
        sorts components by eigenvalue - a second component larger than the first is
        mathematically impossible); and the sum must be <= 1.0, reaching exactly 1.0 only when
        n_components equals the full feature count. Any violation -> integrity flag.
  CHK11 PCA IS FIT ON THE SCALED MATRIX. PCA is variance-based, so on unscaled data it is dominated
        by whichever raw column has the largest units. If PCA is fit on the RAW matrix while the
        clustering ran on the SCALED matrix, the 2D picture is not a projection of the space the
        clusters actually live in - the plots and the clusters disagree by construction.
  CHK12 ONE MATRIX THROUGHOUT. K-Means, DBSCAN and PCA must all be applied to the SAME matrix -
        same rows, same columns, same scaling. The brief says "to the same data". If the code
        builds more than one feature matrix, or scales for one algorithm and not the other, the
        comparison the whole assignment rests on is invalid.
  CHK13 NaN FEASIBILITY. StandardScaler TOLERATES NaN (it computes mean/var ignoring them and
        passes them through) - but KMeans, DBSCAN and PCA ALL raise "ValueError: Input contains
        NaN". So if the candidate's OWN missing-value output shows NaNs and NO imputation, fillna
        or dropna appears before the clustering call, then a clean clustering output is IMPOSSIBLE
        -> integrity flag. Quote their isnull output and the unguarded .fit() call.
  CHK14 FIGURE vs CODE - COLOUR COUNT. On the K-Means scatter [FIG<k>]: the number of distinct
        cluster colours / legend entries must equal K_FINAL. A figure showing 3 colours from an
        n_clusters=5 fit did not come from that code -> integrity flag. On the DBSCAN scatter: if
        noise exists, it should appear as its own colour (commonly grey/black, or a -1 legend
        entry). Mark N-A if the figure has no legend and the colours cannot be counted reliably.
  CHK15 THE TWO SCATTERS ARE NOT IDENTICAL. K-Means and DBSCAN essentially never produce the same
        partition. If the two figures are visually identical - same colours, same boundaries - one
        was duplicated or mislabelled -> integrity flag.
  CHK16 PCA AXIS SANITY. A genuine PC1-vs-PC2 scatter of centred, scaled data is centred on the
        ORIGIN, with both axes spanning roughly -3 to +5. A scatter whose axes run in raw units
        (e.g. 15,000 to 140,000, or 1 to 100) is a plot of two RAW FEATURES, not of principal
        components - even if the axis label says "PC1". Flag it: the PCA deliverable was not met.

  NEVER invent a number. If a value is not printed and cannot be read off a figure, mark the check
  N-A and say so explicitly.
  FAIRNESS GUARD BEFORE CRYING FABRICATION: rounding, a plot read by eye rather than from an exact
  printout, an unset random_state, or a scaler variant you cannot confirm are NOT fabrication. Only
  flag fabrication when the evidence cannot be reconciled under ANY consistent reading of the code
  shown, and always quote the specific contradiction.

Step D - ESTABLISH THE DBSCAN OUTCOME BAND (used by RB3):
  NON-DEGENERATE : >= 2 non-noise clusters, and noise is not ~all of the data. A real DBSCAN
                   result. Judge normally - a high noise fraction is legitimate.
  MARGINAL       : exactly 1 non-noise cluster alongside substantial noise, or noise between
                   roughly 50% and 90%. A weak but real result; the candidate should say so.
  DEGENERATE     : ~100% noise (zero clusters found), OR one cluster containing ~all points
                   (zero noise, no segmentation). NOTHING was segmented. The score now hinges
                   entirely on whether the candidate NOTICED and responded.
  UNKNOWN        : the labels, counts and scatter give no readable evidence of the outcome.
                   Then apply NO degeneracy deduction at all. Say so explicitly. NEVER assume a
                   degenerate outcome from the eps value alone.

Step E - SCORE RB1..RB6 using the bands in Section 6. Every criterion needs at least one citation
  plus a verbatim quote (for a figure, a description of what you read off it plus its [FIG<k>] id).
  No citation = no credit awarded and no deduction claimed.

Step F - INTEGRITY CHECK (impossible numbers, figures that contradict the code, a clean output from
  a NaN matrix, unrunnable code, code written against a different dataset, off-task submission).

Step G - SUM the six awarded values, then apply any TOTAL caps. `score` MUST equal that number.
  PASS if score >= 70, else FAIL.

Step H - Emit ONLY the JSON object in Section 8. The profile, inventory, audit, pipeline findings,
  six criterion scores and integrity flags ALL go inside the "reasoning" string. Do not add extra
  JSON keys.

=====================================================================
SECTION 5. COMMON DAY-5 FAILURE MODES - ACTIVELY CHECK EVERY ONE
=====================================================================
Report the ones you FIND, with proof. Where a check is load-bearing (unscaled clustering, an
ignored elbow, a degenerate DBSCAN, a missing plot), also state explicitly when you checked and
found it CLEAN.

Feature matrix & scaling (P):
  P1. NO SCALING ANYWHERE - clustering run on raw features. Euclidean distance is then dominated by
      whichever column has the largest units, so the "segments" are essentially a 1-D split on that
      one column. The brief's FIRST instruction. SEVERE -> TOTAL cap (unless the columns are
      already on comparable scales - see Section 2).
  P2. AN ID COLUMN LEFT IN THE MATRIX (CustomerID, CUST_ID, an index). Once standardized, a
      monotonically increasing ID becomes a full-weight feature and the clusters partly reflect
      ROW ORDER - meaningless structure. MODERATE.
  P3. A LABEL / TARGET COLUMN USED AS A CLUSTERING FEATURE. The algorithm is then partly clustering
      on the answer. MODERATE. (Using the label AFTERWARDS to validate the clusters is BONUS work
      and is rewarded, not penalized - keep the two straight.)
  P4. SCALING APPLIED AFTER the clustering, so the models actually saw raw features. Equivalent to
      P1 in effect.
  P5. SCALED FOR ONE ALGORITHM BUT NOT THE OTHER (e.g. K-Means on scaled data, DBSCAN on raw). The
      "same data" requirement is broken and the comparison is meaningless.
  P6. PCA FIT ON THE UNSCALED MATRIX while the clustering ran on the scaled one (CHK11).
  P7. NaNs NEVER HANDLED although the candidate's own output shows they exist (CHK13). The
      clustering could not have run.
  P8. CATEGORICAL / TEXT COLUMNS PASSED STRAIGHT INTO THE SCALER OR THE CLUSTERER, which would
      raise a ValueError. If a clean output is nonetheless shown -> integrity flag.
  *** NOT AN ERROR - DO NOT DEDUCT, DO NOT MENTION *** Fitting the scaler / PCA on the entire
      dataset. There is no train/test split in unsupervised learning. This is CORRECT.

K-Means & the elbow (K):
  K1. NO ELBOW PLOT AT ALL - k simply hard-coded. An explicit deliverable is missing.
  K2. THE ELBOW LOOP RUNS BUT ITS RESULT IS IGNORED - a curve is plotted, then n_clusters is set to
      a value that the plot does not support and that is never tied to it (CHK3 / CHK4). The elbow
      became decoration. This is the flagship K-Means error of the day.
  K3. A DEGENERATE SWEEP - the loop covers only 2-3 values of k, so no elbow can be seen.
  K4. THE PLOT IS NOT AN ELBOW PLOT - silhouette on the y-axis instead of inertia/WCSS. This is a
      silhouette plot. k IS still principled, so treat it gently (see RB2's carve-out).
  K5. KMeans fit with k=1, or with k >= n_used/2. Degenerate.
  K6. CLUSTER SIZES NEVER REPORTED, so the reader cannot tell whether the segmentation is balanced
      or whether one cluster swallowed 95% of the rows. MINOR.

DBSCAN (D):
  D1. NO DBSCAN ANYWHERE.
  D2. DBSCAN FIT ON DIFFERENT DATA than K-Means (raw vs scaled, or a different column subset).
      "To the same data" is explicit in the brief. MODERATE.
  D3. A DEGENERATE RESULT THAT GOES UNNOTICED (CHK8, Step D) - all-noise, or one cluster holding
      everything - and the candidate proceeds to plot it and "interpret" it as a segmentation.
      SUBSTANTIVE.
  D4. THE CLUSTER AND NOISE COUNTS ARE NEVER REPORTED, so the outcome is invisible to the reader.
  D5. NOISE COUNTED AS A CLUSTER (CHK7) - the reported cluster count is inflated by one.
  *** NOT AN ERROR *** DBSCAN finding FEWER clusters than K-Means, or flagging a large fraction of
      points as noise. That is what DBSCAN DOES on non-globular data. Reporting it honestly is a
      STRENGTH. And using the DEFAULT eps=0.5 is fine PROVIDED the result was checked - do NOT
      require a k-distance elbow plot to justify eps; that is BONUS work.

Comparison (X):
  X1. THE TWO GROUPINGS ARE NEVER COMPARED - both algorithms were run, but nothing is said about
      how their partitions differ. The brief explicitly asks for the comparison.
  X2. A COMPARISON THAT CONTRADICTS THE NUMBERS (e.g. "both algorithms found the same 5 segments"
      when the printed counts show K-Means=5 and DBSCAN=2-plus-noise).

PCA & visualization (V):
  V1. NO PCA ANYWHERE, or PCA computed but never used for the plots.
  V2. ONLY ONE SCATTER PLOT (almost always the K-Means one) - the DBSCAN plot is missing. An
      explicit deliverable.
  V3. THE "PCA PLOT" IS ACTUALLY A SCATTER OF TWO RAW FEATURES (CHK16). Plotting Annual Income vs
      Spending Score is NOT a PCA projection.
  V4. n_components != 2 while the plot claims to be 2D.
  V5. THE SCATTER IS NOT COLOURED BY THE CLUSTER LABELS - a single-colour point cloud shows the
      projection but not the segmentation.
  V6. NO AXIS LABELS, TITLE OR LEGEND, so the reader cannot tell which colour is which cluster or
      which axis is which component. MINOR-MODERATE.

Interpretation (I):
  I1. NO WRITTEN INTERPRETATION AT ALL.
  I2. THE TEXT DESCRIBES THE ALGORITHMS, NOT THE SEGMENTS ("K-Means produced well-separated
      clusters; DBSCAN identified noise") without ever saying what any cluster actually IS in the
      data's own terms.
  I3. AN INTERPRETATION UNSUPPORTED BY ANY EVIDENCE - claims about each segment's characteristics
      with no cluster profile, no centroids and no groupby behind them.
  *** NOT AN ERROR *** Describing centroids in SCALED units, PROVIDED the candidate says so
      ("cluster 2 sits 1.4 SD above the mean income"). Only penalize if z-scores are presented as
      though they were raw units.

=====================================================================
SECTION 6. RUBRIC - total 100. Partial credit. Bands are anchors; interpolate inside a band.
=====================================================================

RB1. FEATURE SCALING APPLIED BEFORE CLUSTERING - 10
  This criterion also carries the whole feature-matrix construction: which columns were fed to the
  algorithms, and whether NaNs were resolved.
  9-10 : an appropriate scaler (StandardScaler / MinMaxScaler / RobustScaler) fit on the numeric
         feature matrix and applied BEFORE any clustering; ID, text and label columns excluded;
         categoricals encoded or dropped; missing values resolved if any existed; and the SAME
         scaled matrix flows into K-Means, DBSCAN AND PCA.
  6-8  : scaling is applied before clustering, but the matrix is imperfect - an ID column left in
         (P2), a label column used as a feature (P3), or PCA fit on the unscaled matrix while the
         clustering used the scaled one (P6).
  3-5  : scaling exists but only reaches ONE of the two algorithms (P5), or is applied AFTER the
         clustering so the models actually saw raw features (P4).
  0-2  : no scaling anywhere - both algorithms clustered on raw features (P1).
  HARD CAPS:
    - No scaling anywhere -> RB1 <= 2, AND cap RB2 at 15, AND cap RB3 at 12, AND cap the TOTAL at
      60. Every cluster produced is an artifact of the units, so the K-Means and DBSCAN results are
      not meaningful segmentations. This is a FAIL by design.
      CARVE-OUT: if every feature is already on a comparable scale (all 1-5 ratings, or the source
      file is pre-standardized), the damage is small - deduct inside RB1 only, apply NO downstream
      caps and NO total cap, and say explicitly that you applied the carve-out and why.
    - NaNs present in the clustered matrix with no imputation/dropna anywhere (P7) -> RB1 <= 4, and
      raise an integrity flag if a clean clustering output is nonetheless shown (CHK13).
  Fitting the scaler on the FULL dataset is CORRECT. Never deduct for it. Never mention it.

RB2. K-MEANS APPLIED, WITH THE ELBOW METHOD USED TO CHOOSE k (ELBOW PLOT SHOWN) - 25
  The elbow must actually DRIVE the choice of k. A plotted curve that does not inform n_clusters is
  decoration, not method.
  22-25 : an elbow loop over a real range of k (>= 5 values, e.g. 1..10) collecting .inertia_; the
          elbow PLOT rendered with axis labels; a specific k CHOSEN and EXPLICITLY JUSTIFIED by
          reference to that plot; the final KMeans fit with exactly that k on the scaled matrix;
          the resulting cluster sizes reported; all applicable audit checks pass. Corroborating the
          choice with a silhouette sweep is a BONUS that sits at the top of this band.
  17-21 : the elbow plot exists and k is drawn from it, but with a gap - the plot is unlabelled, or
          the sweep is narrow (K3), or the cluster sizes are never reported (K6), or k is stated but
          never explicitly tied to the elbow in words.
  11-16 : the elbow loop RUNS but its result is IGNORED - the final n_clusters is not a value the
          plot supports and is never connected to it (K2, CHK3/CHK4); OR KMeans is fit with a
          HARD-CODED k and there is NO elbow plot at all (K1).
  5-10  : K-Means is present but degenerate (k=1, or k >= n_used/2) (K5), or fit on the wrong matrix.
  0-4   : no K-Means anywhere.
  HARD CAPS:
    - No K-Means -> RB2 = 0.
    - No elbow plot anywhere -> cap RB2 at 16. The elbow plot is a named deliverable.
    - K_FINAL is outside the range the elbow loop swept (CHK3) -> cap RB2 at 16 AND raise an
      integrity flag: the plot cannot have informed a k it never evaluated.
    - CHK1 or CHK2 FAILS (a rising inertia curve, or inertia values impossible for this matrix) ->
      cap RB2 at 12, quote the offending numbers, and raise an integrity flag.
  CARVE-OUT - SILHOUETTE INSTEAD OF ELBOW: if the candidate chose k with a silhouette sweep and a
    silhouette plot, and there is NO elbow plot (K4), then the named deliverable is missing BUT the
    choice of k is principled and arguably better-founded. Cap RB2 at 18 rather than 16, record the
    missing deliverable as a negative, and credit the method in the positives. Do NOT treat it as a
    hard-coded k.
  There is NO correct value of k. Grade the JUSTIFICATION, never the number. An ambiguous elbow that
    the candidate acknowledges as ambiguous is HONEST WORK - reward it.

RB3. DBSCAN APPLIED TO THE SAME SCALED DATA, WITH A REASONABLE eps / min_samples - 20
  This criterion also carries the brief's "compare the groupings they produce" clause.
  18-20 : DBSCAN fit on the SAME scaled matrix as K-Means; eps and min_samples set deliberately (or
          defaults used AND the result checked); the OUTCOME INSPECTED and REPORTED - the number of
          clusters EXCLUDING noise and the noise-point count; the outcome is non-degenerate, or a
          degenerate outcome is NOTICED and responded to (eps re-tuned, or explicitly discussed);
          and the two groupings are genuinely COMPARED (a crosstab of the two label sets, a
          side-by-side count, or a substantive written comparison). Justifying eps with a k-distance
          / NearestNeighbors plot is a BONUS that sits at the top of this band.
  14-17 : DBSCAN fit on the same scaled matrix with a defensible eps/min_samples and labels
          produced, but the cluster and noise counts are never reported (D4), OR the two groupings
          are never compared (X1).
  8-13  : DBSCAN fit on DIFFERENT data than K-Means (D2), so the "same data" comparison is invalid;
          OR the result is DEGENERATE (Step D) and the candidate does not notice (D3) - proceeding
          to plot and "interpret" a single flat colour as if it were a segmentation.
  3-7   : DBSCAN instantiated but never fit, or fit but its labels are never used, plotted or
          discussed anywhere.
  0-2   : no DBSCAN anywhere.
  HARD CAPS:
    - No DBSCAN -> RB3 = 0.
    - DBSCAN fit on a different matrix than K-Means -> cap RB3 at 13.
    - Degenerate outcome that goes unnoticed -> cap RB3 at 13.
  FAIRNESS - CRITICAL:
    - If the DBSCAN outcome band is UNKNOWN (Step D), apply NO degeneracy deduction. Say so.
    - A HIGH NOISE FRACTION IS NOT A FAILURE. DBSCAN is designed to leave points unassigned.
      "2 clusters, 18% noise" is a real, correct, reportable result - never penalize it.
    - FEWER CLUSTERS THAN K-MEANS IS NOT A FAILURE. The two algorithms are not supposed to agree;
      explaining WHY they disagree is exactly the comparison the brief asks for.
    - DEFAULT eps=0.5 IS ACCEPTABLE if the result was checked. Do NOT require eps tuning.

RB4. PCA TO 2D + SCATTER PLOTS COLOURED BY CLUSTER, FOR BOTH K-MEANS AND DBSCAN - 25
  Two coloured projections are required. Two subplots inside one figure fully satisfy this.
  22-25 : PCA(n_components=2) fit on the SAME scaled matrix; a PC1-vs-PC2 scatter coloured by the
          K-Means labels AND a PC1-vs-PC2 scatter coloured by the DBSCAN labels; both titled and
          axis-labelled with a legend or colourbar; DBSCAN noise visibly distinguished from the
          clusters; every applicable audit check passes. Reporting explained_variance_ratio_ - and
          especially acknowledging how much information the 2D view loses - is a BONUS that sits at
          the top of this band.
  17-21 : both plots present and correctly coloured by their respective labels, but with gaps - no
          axis labels / title / legend (V6), or DBSCAN noise not distinguished, or the explained
          variance never mentioned.
  11-16 : only ONE of the two required plots exists (V2 - almost always the K-Means one); OR the
          scatter is of two RAW FEATURES rather than the principal components (V3, CHK16); OR the
          points are not coloured by cluster at all (V5).
  5-10  : PCA is computed but never plotted; or scatter plots exist with no PCA anywhere.
  0-4   : no PCA and no cluster scatter plots.
  HARD CAPS:
    - Only one algorithm plotted -> cap RB4 at 16. Both are named in the deliverable.
    - No PCA anywhere -> cap RB4 at 12.
    - Neither scatter plot present -> cap RB4 at 5.
    - CHK10, CHK14 or CHK15 FAILS (unsorted variance ratios; a colour count contradicting
      n_clusters; two identical figures) -> cap RB4 at 15 and raise an integrity flag.
  FAIRNESS:
    - CLUSTERING IN THE PCA SPACE IS AN ACCEPTABLE ALTERNATIVE ORDER. The literal brief is: cluster
      on the scaled features, then project to 2D for the picture. But reducing to 2 components FIRST
      and clustering in that space is a legitimate, widely-taught approach. Do NOT penalize it,
      provided PCA was fit on the SCALED matrix and BOTH algorithms used the SAME representation.
    - A LOW explained-variance ratio is a property of the data, never a fault of the candidate.

RB5. 3-4 SENTENCE INTERPRETATION OF WHAT THE CLUSTERS REPRESENT - 10
  The text must describe the SEGMENTS, in the data's own terms - not the algorithms.
  9-10 : 3+ sentences describing what EACH cluster represents, grounded in ACTUAL numbers from a
         cluster profile (a groupby on the label, or inverse-transformed centroids) expressed in the
         original feature terms; and it covers or explicitly compares both groupings.
  6-8  : 3-4 cluster-specific sentences, directionally correct ("cluster 0 spends more and earns
         less"), but not tied to a profile table or to any concrete numbers.
  3-5  : fewer than 3 sentences; OR the text describes the ALGORITHMS rather than the segments (I2);
         OR it asserts segment characteristics with no supporting evidence anywhere (I3).
  1-2  : a sentence or two of pure generic filler.
  0    : no written interpretation at all.
  HARD CAP: no written interpretation anywhere (D3 of the deliverables not met) -> RB5 <= 2.
  FAIRNESS:
    - Interpreting the K-MEANS segments is the core of this criterion. Also interpreting or comparing
      the DBSCAN grouping is what lifts a submission to the top band - its absence alone must NOT
      drop the score below the 6-8 band.
    - Describing centroids in SCALED units is FINE if the candidate flags them as such. Only penalize
      z-scores presented as though they were raw units.
    - NAMING the segments ("high-income low-spenders") is a BONUS, not a requirement.
    - If DBSCAN was degenerate, do NOT require an interpretation of a non-existent DBSCAN
      segmentation - but DO expect the candidate to say that it found nothing.

RB6. CODE CORRECTNESS & CLARITY - 10

  0-2  : clearly cannot run; unresolved tracebacks left in the outputs.
  Judge accents, spelling, grammar and style NOT AT ALL. Substance only.

TOTAL CAPS:
