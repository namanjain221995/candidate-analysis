- STEM *OR* LEMMATIZE. The brief says "stem or lemmatize". EITHER ONE ALONE IS FULL CREDIT.

- ANY LINEAR TEXT CLASSIFIER IS FINE: MultinomialNB, ComplementNB, BernoulliNB,

  Then ESTABLISH THE CLASS BALANCE from y.value_counts(), a countplot, or confusion-matrix row
  sums. Record the minority-class share with a citation, and compute the majority-class
  baseline. If it cannot be established from any shown evidence, write "class balance unknown -
  not established from evidence" and apply NO imbalance-related deduction anywhere.

Step B - PIPELINE INVENTORY. With a citation for every item:
  - PREPROCESSING: which of the five named steps appear in code - lowercase, punctuation
    removal, stopword removal, tokenization, stemming OR lemmatization? Which library? In WHAT
    ORDER? Is it a function applied via .apply(), or wired in via the vectorizer's
    preprocessor= / tokenizer= / analyzer= parameter, or an sklearn Pipeline?
  - *** THE CRITICAL TRACE *** WHICH EXACT SERIES IS PASSED TO fit_transform()? Name it. Is it
    the CLEANED column produced by the preprocessing, or the RAW text column? Quote the
    fit_transform call verbatim. This single line decides RB1.
  - VECTORIZER: TfidfVectorizer or CountVectorizer? Which non-default parameters (stop_words,
    max_features, ngram_range, min_df)? Fit on the TRAIN text only, or on the full corpus?
    Is transform() (correct) or fit_transform() (a bug) called on the test text?
  - FEATURE MATRIX: are the shapes printed? X_train shape, X_test shape, vocabulary size.
  - CLASSIFIER: which estimator class? .fit() on X_train? .predict() on X_test? Were the labels
    encoded (LabelEncoder / .map) or left as strings?
  - METRICS: is accuracy reported? is F1 reported? On the TEST set? With what average=? Is a
    confusion matrix or classification_report present (bonus evidence)?
  - TOP WORDS: present? Derived from WHAT - model coefficients (.coef_), Naive Bayes
    log-probabilities (.feature_log_prob_), or a raw corpus frequency count? Mapped back to
    words via get_feature_names_out()? How many lists, and how many words per list?

Step C - LEAKAGE HUNT + AUDIT. Work Section 5 top to bottom. Then run every check below on the
  code, numbers and words you can actually SEE. Report each as PASS / FAIL / N-A with evidence.

  CHK1  VOCABULARY / MATRIX SHAPE CONSISTENCY. If a feature-matrix shape is printed, its column
        count must equal len(vectorizer.vocabulary_) / len(get_feature_names_out()). If
        max_features=N was set, the column count must be <= N (and exactly N whenever the corpus
        has at least N distinct terms). A printed matrix with more columns than max_features
        allows is impossible.

  CHK2  *** TRAIN/TEST COLUMN PARITY - A HARD IMPOSSIBILITY CHECK ***
        X_train and X_test MUST have the SAME number of columns; the model was fit on the train
        column space and predicts in that space. If BOTH shapes are printed and the column counts
        DIFFER, then .predict(X_test) would have raised
        "X has <n> features, but <Model> is expecting <m> features as input" - so a clean
        prediction and a clean accuracy CANNOT coexist with mismatched shapes -> integrity flag.
        The usual cause is calling fit_transform() on the TEST text instead of transform(), which
        re-fits the vectorizer and builds a different vocabulary with a different column ordering
        (see F2). Even when the counts coincidentally match, the columns then MEAN different
        words and the predictions are garbage - so a HIGH accuracy is also not reachable.

  CHK3  *** WHICH COLUMN REACHED THE VECTORIZER - THE FLAGSHIP DAY-6 CHECK ***
        Trace the exact argument of fit_transform(). If a clean_text() / preprocess() function
        exists and produces df['clean_text'], but fit_transform is called on df['review'] (the
        RAW column), then THE ENTIRE PREPROCESSING PIPELINE IS DEAD CODE. It was written,
        displayed, and never used. Quote BOTH the function definition and the fit_transform call.
        This is the single most common serious error on Day 6. Corroborate it with CHK4.
        NOT AN ERROR: wiring the function in via TfidfVectorizer(preprocessor=clean_text) or
        tokenizer= or analyzer=, or inside an sklearn Pipeline. Those DO reach the vectorizer -
        credit them fully.

  CHK4  *** THE TOP-WORDS TABLE AS PROOF OF PREPROCESSING - THE DAY-6 SIGNATURE CHECK ***
        The words in the top-words output came out of the vectorizer's vocabulary, so they are
        direct, independent evidence of what the vectorizer actually received:
        (a) STOPWORDS PRESENT ("the", "and", "i", "it", "to", "is", "this", "a", "of") despite a
            claimed stopword filter -> the filter NEVER REACHED THE VECTORIZER. Hard proof.
            Cross-check CHK3. Quote the offending words.
        (b) STEMS PRESENT ("movi", "wast", "terribl", "disappoint", "recommend", "veri") ->
            stemming DID run and DID reach the vectorizer. Credit it explicitly.
        (c) FULLY-INFLECTED WORDS ("movies", "watching", "loved", "terrible") despite a claimed
            PorterStemmer / SnowballStemmer -> stemming did NOT reach the vectorizer. Hard proof.
        (d) STOPWORD-STEMS PRESENT ("thi", "wa", "hi", "ani", "veri", "onli", "becaus", "hav")
            -> the pipeline STEMMED BEFORE REMOVING STOPWORDS, so the unstemmed stopword list no
            longer matched the stemmed tokens and the stopwords survived. See CHK15 / P2.
        (e) "br" present -> raw HTML tags (<br />) were never stripped. MINOR - note it.
        *** FAIRNESS - CRITICAL *** DO NOT use this check against a LEMMATIZATION claim.
        WordNetLemmatizer leaves most tokens visually unchanged without a POS tag
        (lemmatize("watching") == "watching"), so normal-looking words are NOT evidence that
        lemmatization was skipped. Mark (c) N-A whenever the claim is lemmatization rather than
        stemming.
        *** ALSO *** DO NOT use capitalisation against a lowercase claim. CountVectorizer and
        TfidfVectorizer set lowercase=True BY DEFAULT, so feature names are lowercase regardless
        of whether the candidate called .lower(). Mark that sub-check N-A.

  CHK5  *** LOGISTIC REGRESSION coef_ SHAPE - A HARD IMPOSSIBILITY CHECK ***
        For a BINARY target, sklearn's LogisticRegression stores ONE coefficient row:
        coef_.shape == (1, n_features), NOT (2, n_features).
        Therefore, on a BINARY problem, code of the form
            for i, cls in enumerate(model.classes_): coefs = model.coef_[i]
        raises "IndexError: index 1 is out of bounds for axis 0 with size 1" on the second
        iteration. If such a loop is shown AND a clean two-class word list is nonetheless
        printed, THAT OUTPUT CANNOT COME FROM THAT CODE -> integrity flag. Quote both.
        For a MULTICLASS target (>2 classes) coef_ IS (n_classes, n_features), so the same loop
        is CORRECT -> mark PASS.
        NOTE: MultinomialNB is different. Its feature_log_prob_ IS (n_classes, n_features) even
        for binary, so indexing [0] and [1] is valid there. Do not confuse the two.

  CHK6  TOP-WORDS DIRECTION / SIGN. For a BINARY LogisticRegression, the single coefficient row
        is oriented toward classes_[1]: the most POSITIVE coefficients are the top words for
        classes_[1], and the most NEGATIVE coefficients are the top words for classes_[0].
        Now READ THE ACTUAL WORDS. If the list presented under the NEGATIVE / ham / not-
        recommended class is full of overtly positive terms ("great", "excellent", "perfect",
        "love", "best", "wonderful") while the POSITIVE class list is full of overtly negative
        terms ("worst", "waste", "boring", "awful", "refund"), THE SIGN IS INVERTED and the two
        class lists have been swapped. This is a semantic read you are fully capable of making.
        The artifact is then actively misleading. Flag it as a substantive error.

  CHK7  TOP-WORDS SOURCE. The deliverable is the top PREDICTIVE words per class. That means:
        derived FROM THE FITTED MODEL (LogisticRegression .coef_ or MultinomialNB
        .feature_log_prob_), and mapped back to real words through get_feature_names_out() or
        .vocabulary_. NOT the deliverable: the corpus's most FREQUENT terms (a global
        CountVectorizer column sum), a wordcloud of all the text, or a per-class raw frequency
        count. Those come from the DATA, not the MODEL - they tell you what is common in each
        class, not what DISCRIMINATES the classes. Also not the deliverable: printing raw feature
        INDICES instead of words.

  CHK8  MULTINOMIALNB REQUIRES NON-NEGATIVE FEATURES - A HARD IMPOSSIBILITY CHECK.
        TF-IDF and BoW are both non-negative, so MultinomialNB fits them fine. BUT if a
        StandardScaler was applied to the feature matrix first, the values become negative and
        MultinomialNB raises "ValueError: Negative values in data passed to MultinomialNB
        (input X)". Separately, StandardScaler on a SPARSE matrix with the default
        with_mean=True raises "ValueError: Cannot center sparse matrices. Pass `with_mean=False`
        instead." If either pattern appears in the code and a CLEAN fit output is nonetheless
        shown -> integrity flag.
        (Scaling a TF-IDF matrix is unnecessary anyway - it is already L2-normalized. Its
        ABSENCE is CORRECT and is never a negative.)

  CHK9  *** F1 AVERAGING FEASIBILITY - A HARD IMPOSSIBILITY CHECK ***
        sklearn's f1_score defaults to average='binary', pos_label=1.
        (a) MULTICLASS target (>2 classes) + a bare f1_score(y_test, y_pred) with no average= ->
            raises "ValueError: Target is multiclass but average='binary'. Please choose another
            average setting". A clean numeric F1 printed alongside such a call is IMPOSSIBLE ->
            integrity flag.
        (b) BINARY target with STRING labels ('spam'/'ham', 'positive'/'negative') that were
            NEVER ENCODED to 0/1, plus a bare f1_score with no pos_label= -> raises
            "ValueError: pos_label=1 is not a valid label. It should be one of ['ham', 'spam']".
            A clean numeric F1 printed alongside such a call is IMPOSSIBLE -> integrity flag.
        (c) If the labels WERE encoded (LabelEncoder, or .map({'ham':0,'spam':1})), or an
            explicit average= / pos_label= was passed, the call is fine -> PASS.
        This check is worth running carefully. It is one of the most common places a fabricated
        or hand-typed metric betrays itself.

  CHK10 METRIC RECONCILIATION (only if a confusion matrix or classification_report is printed -
        both are BONUS, but they are free evidence). For a binary CM = [[TN, FP], [FN, TP]]:
        - the cells must sum to len(y_test), which must be approximately n_used * test_size;
        - Accuracy = (TP + TN) / total;
        - F1 = 2*TP / (2*TP + FP + FN);
        - F1 must lie between min(Precision, Recall) and max(Precision, Recall).
        If the printed metrics cannot be reconciled with the printed CM under ANY consistent
        reading, the numbers are fabricated -> integrity flag. Quote them.

  CHK11 PERFECT / NEAR-PERFECT SCORES. Accuracy or F1 at or above ~0.99 on a text-sentiment task
        is a near-certain LEAKAGE signal, not an achievement. Look for a NON-TEXT column that
        encodes the label sitting inside the feature matrix (see F5): a star rating alongside a
        rating-derived sentiment label, a "negativereason" column alongside a sentiment label, a
        confidence column, a recommendation flag. Cross-check and, if found, apply the TOTAL cap.
        (SMS spam is a genuine exception - a well-fit spam classifier legitimately reaches ~0.98.
        So investigate; do not assume.)

  CHK12 MAJORITY-CLASS BASELINE. baseline = largest class count / total. If the model's accuracy
        is approximately the baseline AND the minority-class F1 or recall is near 0, it is a
        majority-class predictor with no real skill. Report it. This is NOT automatically a
        deduction - it becomes one only if the candidate presents it as a success.

  CHK13 ROW-COUNT CONSISTENCY. If sampling was used, the matrix row counts must reflect the
        SAMPLED size, not the raw file size. X_train.shape[0] + X_test.shape[0] must equal
        n_used, and X_train.shape[0] must equal len(y_train).

  CHK14 CLASS COUNT vs NUMBER OF WORD LISTS. The number of top-words lists must equal the number
        of classes. A 3-class target (positive / neutral / negative) with only two word lists is
        an incomplete deliverable.

  CHK15 STOPWORD / STEM ORDER BUG. If the code STEMS FIRST and REMOVES STOPWORDS AFTER, the
        stopword list (which contains unstemmed words) no longer matches the stemmed tokens, so
        the stopwords survive as stems. Same bug: passing stop_words='english' to the vectorizer
        on text that has ALREADY been stemmed - sklearn's list contains "this", not "thi", so it
        filters nothing. Evidence: stopword-stems ("thi", "wa", "hi", "ani", "veri", "onli",
        "becaus") appearing in the top-words table. A real, subtle, verifiable pipeline bug.
        The correct order is: tokenize -> remove stopwords -> stem/lemmatize.

  NEVER invent a number, a word, or a shape. If something is not printed anywhere, mark the check
  N-A and say so explicitly.
  FAIRNESS GUARD BEFORE CRYING FABRICATION: rounding, an unset random_state, a truncated output,
  or a library version you cannot confirm are NOT fabrication. Only flag fabrication when the
  evidence cannot be reconciled under ANY consistent reading of the code shown, and always quote
  the specific contradiction.

Step D - ESTABLISH THE IMBALANCE CONTEXT from Step A (used for commentary in RB4 only):
  minority share >= 35%  -> BALANCED. Accuracy is informative. No comment needed.
  15% - 35%              -> IMBALANCED. F1 is the more informative metric. The brief REQUIRES F1

  < 15%                  -> SEVERELY IMBALANCED. Accuracy is nearly meaningless; F1 and the

  unknown                -> apply NO imbalance-related comment or deduction. Say so explicitly.
  NOTE: unlike Day 3, F1 is a NAMED DELIVERABLE here. So a missing F1 is simply a missing

Step E - SCORE RB1..RB6 using the bands in Section 6. Every criterion needs at least one citation

Step F - INTEGRITY CHECK (impossible outputs per CHK2/CHK5/CHK8/CHK9, metrics that contradict a

=====================================================================
SECTION 5. COMMON DAY-6 FAILURE MODES - ACTIVELY CHECK EVERY ONE
=====================================================================
Report the ones you FIND, with proof. Where a check is load-bearing (dead preprocessing, vectorizer
leakage, a missing F1, a missing or misleading word table), also state explicitly when you checked
and found it CLEAN.

Preprocessing pipeline (P):
  P1. *** THE FLAGSHIP DAY-6 ERROR *** A clean_text() function is written, applied to create a new
      column, and then the RAW text column is what gets passed to fit_transform. The whole
      preprocessing pipeline is DEAD CODE - written, displayed, never used (CHK3). Corroborate
      from the top-words table (CHK4a): if the filter never ran, the stopwords will be sitting
      right there in the top-10.
  P2. Stemming applied BEFORE stopword removal, so the unstemmed stopword list no longer matches
      the stemmed tokens and stopwords survive as stems (CHK15, CHK4d).
  P3. stop_words='english' passed to the vectorizer on text that has ALREADY been stemmed - the
      sklearn list contains "this", not "thi", so it filters nothing. Same root cause as P2.
  P4. A named step is simply ABSENT - no stemming or lemmatization anywhere; no stopword removal
      anywhere and no stop_words parameter either.
  P5. Punctuation stripped with a method that also destroys the token boundaries (removing all
      non-alphabetic characters INCLUDING whitespace), collapsing each document into a single
      enormous token. Symptom: a vocabulary size roughly equal to the document count. Fatal.
  P6. HTML tags never stripped, so "br" (from <br />) becomes a high-weight feature. Evidence:
      "br" in the top-words table. MINOR - deduct at most 1-2.
  P7. The TRAIN text and the TEST text are preprocessed DIFFERENTLY (two functions, or one applied
      and the other not), so the two matrices are not in the same representation.

Feature extraction & split (F):
  F1. THE VECTORIZER IS FIT ON THE FULL CORPUS BEFORE THE SPLIT:
          X = vec.fit_transform(all_text);  train_test_split(X, y, ...)
      The vocabulary and the IDF statistics were computed with the test documents included, so
      information from the test set entered the representation and the reported metrics are
      optimistically biased.
      *** CALIBRATE THIS CORRECTLY *** This is a REAL methodological error and deserves a real
      deduction in RB2. But it leaks the DISTRIBUTION, not the LABEL. The metrics are BIASED, not
      WORTHLESS. Do NOT apply a TOTAL cap for it and do NOT treat it like target-in-X leakage.
      The correct pattern is to split the TEXT first, then vec.fit_transform(X_train_text) and
      vec.transform(X_test_text) - or to wrap both in an sklearn Pipeline.
  F2. fit_transform() called on the TEST text instead of transform(). This RE-FITS the vectorizer,
      producing a different vocabulary and a different column ordering, so the model is predicting
      in a column space that does not mean what it was trained on. sklearn will usually raise
      outright on the feature-count mismatch (CHK2). A clean, HIGH accuracy is not reachable from
      this code.
  F3. NO TRAIN/TEST SPLIT ANYWHERE - the model is trained and scored on the same rows, so every
      metric is in-sample and tells you nothing about generalization.
  F4. train_test_split applied to the MATRIX rather than to the TEXT. This is the mechanical cause
      of F1 - name it as such when you see it.
  F5. *** SEVERE *** A NON-TEXT COLUMN THAT ENCODES OR IS DERIVED FROM THE LABEL LEFT INSIDE THE
      FEATURE MATRIX - a star rating alongside a rating-derived sentiment label; a
      "negativereason" column alongside a sentiment label; a confidence score; a recommendation
      flag. The classifier is then reading the answer. Symptom: accuracy near 1.00 (CHK11).
      -> TOTAL cap. (Reminder: USING such a column to DERIVE the label is correct. Only leaving it
      in the FEATURES is the error.)
  F6. The feature-matrix shape is never printed, so the vocabulary size is invisible to the
      reader. MINOR.

Modelling (M):
  M1. NO CLASSIFIER TRAINED. Sentiment produced by a LEXICON scorer (VADER, TextBlob) or a
      pre-trained pipeline instead. There is then no feature matrix, no fitted coefficients and no
      top predictive words - RB2, RB3 and RB5 all collapse. OFF-TASK -> TOTAL cap.
